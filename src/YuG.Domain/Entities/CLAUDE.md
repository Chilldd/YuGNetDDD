# DDD 领域建模核心规范

## 一、核心设计原则

- **以业务行为为中心**：先建模行为（做什么），再建模数据（有什么）
- **聚合保证一致性**：聚合根是唯一修改入口，始终维护业务不变性（Invariant）
- **领域层纯净**：不依赖 ORM / 框架 / 基础设施（POCO）
- **边界清晰**：一个聚合只负责一个一致性边界，避免“万能对象”
- **统一语言**：命名体现业务语义，避免技术导向命名（如 Update、Process）
- **领域事件边界清晰**：仅用于跨聚合或跨限界上下文的业务事实传播

---

## 二、必须事项（DO）

- 行为建模
  - 使用业务语义方法
    - `Rename()`
    - `Disable()`
    - `MoveTo()`
    - `Disable()`
    - `ChangeEndpoint()`
  - 方法必须表达“业务动作”，而不是字段修改
- 封装性
  - 属性使用 `private set`
  - 所有状态变更必须通过方法完成
  - 禁止外部直接修改领域对象状态
- 业务规则内聚
  - 在领域模型内部完成业务校验
  - 保证任何时刻对象都处于合法状态（Invariant）
- 聚合设计
  - 聚合必须以“强一致性规则”为边界
  - 一个事务内只保证一个聚合的一致性
  - 跨聚合一致性使用领域事件或最终一致性
- 领域事件使用规范
  - 仅用于“业务事实已发生”
  - 触发条件：
    - 需要其他限界上下文响应
    - 或需要异步业务处理
  - 禁止滥用（如每个字段变化都发事件，或者还没开始做的业务）
- 方法语义一致性
  - 信息变更 → `ChangeXxx` / `Rename`
  - 结构变更 → `MoveTo` / `ChangeEndpoint`
  - 状态变更 → `Activate` / `Disable`
  - 禁止混合语义方法（如 `UpdateBasicInfo`）
- 复杂逻辑下沉
  - 业务规则优先下沉到领域模型
  - Application Service / Handler 只负责协调流程
  - 跨聚合规则使用 Domain Service

---

## 三、禁止事项（DON’T）

- 禁止 `public set`
- CRUD风格方法
  - 禁止：
      - `UpdateXxx`
      - `SetXxx`
      - `Update(...)`
  - 原因：
      - 无法表达业务意图
      - 不利于规则扩展
      - 会退化为数据模型
- 贫血模型
  - 禁止只有属性 + getter/setter
  - 禁止无行为的实体
- 绕过聚合修改数据
  - Repository 不允许直接修改字段
  - 外部不能操作子实体
- 领域层依赖基础设施
  - 禁止依赖：EF Core, MediatR, DB / ORM / MQ, HTTP / 外部服务
- 大而全聚合
  - 一个聚合包含多个业务职责
  - 导致修改范围过大、维护困难
- 过度值对象化
  - 不要为了“DDD看起来高级”而强行拆值对象
  - 只有具备独立语义和规则的概念才应该抽象
- 滥用领域事件
  - 禁止所有状态变化都发布事件
  - 禁止用于日志、审计、UI通知等弱业务场景
  - 禁止未开发的业务提前设计领域事件

---

## 四、状态机与业务一致性
- 状态变更必须符合业务状态流转规则
- 禁止任意 SetStatus
- 必须定义合法状态迁移路径
- 不允许跳跃式状态修改
