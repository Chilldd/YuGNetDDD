# 领域服务（Domain Service）开发规范

## 核心设计原则

**应该使用领域服务**：
- 业务流程涉及多个实体或聚合
- 逻辑不自然地属于某一个实体
- 需要访问仓储来检查业务规则
- 算法类的业务操作（如推荐引擎、价格计算等）

**不应使用领域服务**：
- 仅操作单个实体（使用实体方法）
- 纯粹的基础设施操作（使用应用服务）
- 业务逻辑可以清晰地属于某个实体

---

## 必须事项

- 领域服务定义在 `YuG.Domain/Services/`
- 命名规范：`I{ServiceName}Service` 接口 + `{ServiceName}Service` 实现
- 领域服务必须是**无状态的**，不持有可变域
- 同一实例可以被多个线程/请求并发使用
- 所有的输入参数通过方法参数传递，所有结果通过返回值返回
- 领域服务应定义接口，便于测试和替换
- 接口定义在 Domain 层，实现也在 Domain 层（无外部依赖）或 Infrastructure 层（如依赖外部服务）


## 禁止事项

- **禁止有可变状态**，所有状态应通过参数传入和返回值返回
- **禁止产生领域事件**，事件只在聚合根中产生
- **禁止返回 DTO**，本服务只处理 Domain 对象
- **禁止在 Infrastructure 层调用**，仅 Application 和 Domain 调用
- **禁止依赖注入可变状态**，通过方法参数传递
- **禁止互相依赖**，单一职责原则
